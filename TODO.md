### Режимы работы (как у рулетки)

Управляется галочкой **«Авто-скачки»** в админке (`auto_enabled` в SQLite).

| `auto_enabled` | Кто открывает подготовку | Кто запускает забег |
|----------------|--------------------------|---------------------|
| **Вкл** | Первая команда `!скачки` в чате | Таймер «Подготовки» (сбор ставок) |
| **Выкл** | Кнопка «Открыть ставки» в админке | Кнопка «Старт забега» в админке |

**Машина состояний:**

```
IDLE → OPEN → RACE_WAIT → RACE → COOLDOWN → IDLE
```

| Состояние | Поведение |
|-----------|-----------|
| **IDLE** | Ставки не принимаются (ручной режим) или ждут `!скачки` (авто) |
| **OPEN** | Фаза **«Подготовка»**: принимаются `!скачки <сумма> <номер>` после просмотра состава |
| **RACE_WAIT** | Ставки закрыты, пауза перед симуляцией (анимация в OBS, как `SPIN_WAIT` у рулетки) |
| **RACE** | Симуляция забега, события на трассе, финиш, расчёт выплат |
| **COOLDOWN** | Глобальный откат, новые ставки отклоняются |

**Авто-режим (`auto_enabled = true`):**
1. Игрок пишет `!скачки` → раунд → `OPEN`, сформирован состав из 10 принцесс, состав показан в чате.
2. Игроки делают ставки `!скачки <сумма> <номер>`, видя состав.
3. По истечении таймера: «Ставки больше не принимаются!» → `RACE_WAIT` → `RACE` → выплаты → `COOLDOWN`.

**Ручной режим (`auto_enabled = false`):**
1. `!скачки` в `IDLE` → «Забег закрыт. Ждите запуска от стримера».
2. Админ открывает ставки → `OPEN`.
3. Чат ставит в окне сбора.
4. Админ жмёт «Старт забега» (или таймер) → `RACE_WAIT` → `RACE` → `COOLDOWN`.

**Сообщения в чат:** не отвечать на каждую успешную ставку — только открытие, закрытие ставок, ключевые события забега (опционально, сжато), финиш и выплаты. **Исключение:** отклонённая повторная ставка.

### Выплаты: коэффициент, не множитель шанса

В отличие от рулетки, выигрыш **не** считается как «ставка × (36/N)» от вероятности сектора.

**Идея:** у каждой «лошади» (принцессы) в раунде есть **коэффициент** (odds). При победе:

```
Выплата = ⌊ставка × коэффициент⌋
```

Коэффициент **динамический** и должен отражать:
- **Популярность в текущем раунде** — чем больше сумма ставок на эту лошадь относительно общего пула, тем **ниже** коэффициент.
- **Историю побед** — чем чаще принцесса побеждала в прошлых забегах, тем **ниже** коэффициент.

Чем чаще на неё ставят **и** чем чаще она выигрывает → тем меньше получишь, если поставил на неё и она снова победила.

**Черновая формула** (уточнить константы в `settings.py` при реализации):

```python
# На момент закрытия ставок (или финальный снимок перед RACE):
bet_share[i] = сумма_ставок_на_i / max(1, общая_сумма_ставок_раунда)   # 0..1
win_rate[i]  = побед_принцессы / max(1, забегов_с_участием)            # 0..1, из SQLite

base = BASE_ODDS   # например 9.0 для поля из 10 (аналог «средних» odds)

raw[i] = base / (1 + ALPHA * bet_share[i] + BETA * win_rate[i])
coefficient[i] = clamp(raw[i], MIN_COEFFICIENT, MAX_COEFFICIENT)
```

- `ALPHA`, `BETA` — веса популярности и истории (подобрать на тестах).
- Коэффициент **фиксируется** на момент закрытия ставок (`OPEN` → `RACE_WAIT`), чтобы игроки видели odds до финиша.
- В админке и API показывать таблицу: №, принцесса, коэффициент, сумма ставок на неё.

**Казна:** общая `minigames_bank` — ставки в казну, выплаты из казны, пропорциональное урезание при банкротстве, `MIN_BANK_TO_START`, блокировка обеих игр при пустой казне.

### Симуляция забега

После `RACE_WAIT` запускается пошаговая (или тиковая) симуляция:

1. 10 участниц стартуют с позиции 0.
2. На каждом тике у каждой — случайный прирост дистанции + модификаторы от событий.
3. Первые пересекают финиш — определяется место (для выплат важен только **победитель**).
4. Результат сохраняется в `last_result` (победитель, состав, события, выплаты).

**Случайные события** (примеры, список расширить в `simulate.py`):

| Событие | Эффект |
|---------|--------|
| Открылось второе дыхание | +скорость на несколько тиков |
| Споткнулась и упала | пауза / откат позиции |
| Срезала угол | краткий бонус |
| Запуталась в гриве | замедление |
| Фанаты подбадривают | небольшой бонус |
| Судьёжка | нейтральное флavour-сообщение в чат (без механики) |

События должны быть **зрелищными** для OBS-оверлея: API отдаёт текущие позиции и последнее событие для анимации.
